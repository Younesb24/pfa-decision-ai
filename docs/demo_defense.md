## 4. The two AI pillars — how each works in 30 seconds

### Pillar A — XGBoost late-delivery prediction

- **Model:** XGBClassifier, 34 features (seller history, order details, geography, time, payment), trained on `gold.fct_orders` with time-based train/test split.
- **Class imbalance:** handled via `scale_pos_weight` (~11×), not SMOTE.
- **Performance:** ROC-AUC ≈ 0.82, F1 ≈ 0.38 after threshold tuning to optimize F1 with recall ≥ 0.50. Defended as a **risk-prioritization tool**, not a hard classifier.
- **Explainability:** lightweight `importance × deviation` heuristic, not SHAP. Documented as such in the modal footer ("Main risk signals") and the model card.
- **What clicking a seller does:**
  1. Backend pulls seller history from `gold.agg_seller_scorecard` and state from `gold.dim_seller`.
  2. Builds a feature vector using marketplace reference values for the order-level features (this is the "typical next order" assumption).
  3. Runs `predict_proba`, returns probability + threshold + top-3 contributing factors + concrete recommendation.
  4. Computes a separate `historical_risk_label` from the scorecard composite so the UI can show the divergence honestly.

### Pillar B — LLM Ask Bar (text-to-SQL)

- **Provider:** OpenAI `gpt-4o` (configured via `OPENAI_API_KEY` in `api/.env`). Anthropic Claude is a supported alternative.
- **Semantic layer:** a detailed system prompt documenting every Gold table and column.
- **Safety layers:**
  1. Model instructed to emit only `SELECT`/`WITH` against `gold.*`.
  2. Output stripped of code fences and rejected if it doesn't start with `SELECT`/`WITH`.
  3. Access to `pg_catalog` / `pg_*` / `information_schema` rejected.
  4. Multi-statement input rejected.
  5. Result rows capped at 50.
- **Audit:** every request — success or rejection — is logged to `governance.audit_log`.

## 5. Demo questions for the Ask Bar (verified working)

| Question | What it proves |
|---|---|
| `Why did OTIF drop in August 2018?` | Joins `agg_daily_ops_kpi` with `dim_date`, returns daily OTIF for August — shows the LLM can navigate the semantic layer |
| `Which sellers should I prioritize and why?` | Uses `agg_seller_scorecard`, ranks by composite score — shows the LLM understands the seller risk model |
| `Which categories generated the most revenue in August 2018?` | Joins `fct_orders` × `dim_product` × `dim_date`, groups by category — shows multi-table reasoning |
| `Which sellers had the highest late delivery rate this year?` | Filters `agg_seller_scorecard` by year via `fct_orders` — good backup question |

If a question fails, the safety layers reject it cleanly with an error — that's also defensible: *"the system refuses to execute unsafe SQL."*

## 8. Ingest — what happens when you drop a CSV

1. File is validated (must be .csv, max 200 MB).
2. Server reads the bytes, profiles the schema (column names, dtypes, null counts, sample values).
3. A row is inserted into `governance.source_registry` with the schema profile as JSON.
4. The file is written to `data/ingest/<source_id>/raw.csv`.
5. A suggested dbt staging table name is returned.

**What it does NOT do:**
- Does NOT auto-create a dbt model.
- Does NOT update the dashboard.
- Does NOT feed the AI surfaces.

**Demo guidance:** **Do not show ingest in the main 3-minute flow.** Mention it only if asked about extensibility:

> *"This screen is the data-source onboarding surface — drop a CSV, the server profiles its schema and registers it for a follow-up dbt scaffolding step. It's intentionally not part of the demo flow because the decision use case is demonstrated on the Olist dataset. The architecture is designed to extend to API/ERP connectors, but those are post-MVP."*

If the jury insists on seeing it: drop a small CSV (e.g. a 5-row test file), show the schema preview, mention the registry row. Don't promise the dashboard will update.

## 10. Ten likely jury questions — short defense answers

1. **"Where is the AI exactly?"**
   *"Two surfaces. A trained XGBoost classifier — you saw it predict the late-delivery probability for that seller and explain the risk signals. And an LLM Ask Bar that converts natural-language questions into safe SQL against the Gold layer. Everything else — KPIs, anomaly detection, dashboards — is data engineering that feeds those two AI surfaces."*

2. **"Why not use ChatGPT directly?"**
   *"ChatGPT doesn't know my data. My system feeds the LLM only pre-computed KPIs from the Gold layer — the LLM never touches raw data, never runs uncontrolled SQL, never invents numbers. A self-critique pass fact-checks every claim. Strip the LLM and the dashboard still works — the LLM is a narration layer, not the engine."*

3. **"How accurate is your model?"**
   *"ROC-AUC around 0.82 on a time-based test split. F1 is around 0.38 because the positive class is only 8%. We defend it as a risk-prioritization tool — it ranks risky orders above non-risky ones reliably — not as a hard classifier. The operator uses it to choose who to contact this week, not to auto-block sellers."*

4. **"Why XGBoost and not a neural network?"**
   *"100K orders with mostly tabular features. Gradient boosted trees outperform neural networks on this regime, train in seconds, and give interpretable feature importances. Neural nets would add complexity without measurable accuracy gain."*

5. **"How did you use customer reviews?"**
   *See section 6.*

6. **"Why did the AI briefing say a number that wasn't right?"** (if self-critique flags something during demo)
   *"That's exactly what the self-critique pass exists to catch — the LLM tried to cite a number not in the data context, and the system flagged it. This is the 'narrator, never calculator' principle in action."*

7. **"Why not LangChain / RAG / vector DB?"**
   *"My corpus is the KPI catalog and data dictionary — two short documents. Stuffing them into a system prompt works and is debuggable. A vector DB would be justified above a thousand documents; here it would add latency and a moving part without capability. There is an ADR documenting this decision."*

8. **"Why z-score for anomalies and not Isolation Forest?"**
   *"Z-score is interpretable — the operator can read 'OTIF is 2.2 standard deviations below the mean'. The value of the system is not the detection algorithm; it's the full loop: detect → diagnose with AI → act → track outcome. Z-score is the trigger, not the answer."*

9. **"How does this work for a different marketplace?"**
   *"Olist is the demonstration case. The architecture is reusable: dbt medallion warehouse, persona-aware briefings, anomaly detection, XGBoost ranking, audit trail. Replace the source CSVs with another marketplace and re-run dbt. The ingest endpoint exists for that future."*

10. **"Did you use AI to build this?"**
    *"I used AI as a coding assistant — the same way a developer uses GitHub Copilot or Stack Overflow. The architectural decisions are mine: the OODA framing, the medallion warehouse, the 'LLM as narrator' principle, the choice of XGBoost over neural networks. I can defend every line because I wrote the specification and validated every component."*
